# 项目代码风格记录

本文档基于当前仓库实际代码抽样整理，目标是记录项目中已经形成的代码风格，便于后续新增代码时尽量保持一致。

## 1. 总体判断

- 后端整体风格以若依系 Spring Boot / Spring Cloud 多模块项目为主，分层结构稳定，命名和接口组织方式较统一。
- 前端整体风格以 Vue 2 + Element UI 的后台管理模板为基础，业务页面在模板风格上做了较明显的视觉增强。
- 项目没有发现统一的 `checkstyle`、`spotless`、`.prettierrc`、`.eslintrc` 等强约束配置，因此当前风格更多依赖既有代码习惯，而不是自动格式化规则。
- 前端存在 `.editorconfig`，说明前端至少在缩进、换行和尾随空格方面有明确约定。

## 2. 目录与分层风格

### 后端

- 采用标准微服务拆分：
  - `edu-auth` 认证中心
  - `edu-gateway` 网关
  - `edu-api` 远程接口
  - `edu-common` 公共能力
  - `edu-modules` 业务模块
  - `edu-visual` 监控模块
- 业务模块内部基本遵循 `controller`、`service`、`mapper`、`domain`、`config` 的分层方式。
- 常见调用链为：`Controller -> Service -> Mapper -> XML SQL`。
- 公共响应、分页、权限、日志等能力统一复用公共模块，不倾向于在业务模块重复造轮子。

### 前端

- 采用典型后台系统目录组织：
  - `api` 放接口封装
  - `views` 放页面
  - `components` 放复用组件
  - `layout` 放壳层
  - `utils` 放工具
  - `plugins`、`directive` 放全局扩展
- 页面通常按业务域拆分，例如 `views/achievement/...`、`api/achievement/...`。

## 3. 后端 Java 风格

### 3.1 命名习惯

- 包名全小写，采用业务域前缀，如 `com.edu.achievement`、`com.edu.system`。
- 类名使用 PascalCase，例如 `EduAchievementController`、`SysUserController`。
- Service 接口保留 `I` 前缀，例如 `IEduAchievementService`、`ISysUserService`。
- Mapper 接口与 XML 一一对应，名称直接使用实体名加 `Mapper`。
- 实体类与表名语义对应，通常继承 `BaseEntity`。

### 3.2 分层写法

- Controller:
  - 统一使用 `@RestController` + `@RequestMapping`。
  - 大多继承 `BaseController`。
  - 列表接口调用 `startPage()`，返回 `getDataTable(...)`。
  - 增删改接口常返回 `AjaxResult`，并通过 `toAjax(...)` 包装结果。
  - 权限控制通过 `@RequiresPermissions`。
  - 审计日志通过 `@Log(title = "...", businessType = ...)`。
- Service:
  - 接口定义能力边界，实现类命名为 `XxxServiceImpl`。
  - 实现层多数只保留业务逻辑和少量时间字段处理，例如 `createTime`、`updateTime`。
- Mapper:
  - 使用 MyBatis 接口 + XML。
  - 查询条件通过 XML 动态 SQL 组织，不倾向于注解 SQL。

### 3.3 代码组织与格式

- Java 文件普遍使用 4 空格缩进。
- 类的大括号采用另起一行的 K&R/RuoYi 混合风格，例如：

```java
public class EduAchievementController extends BaseController
{
    @Autowired
    private IEduAchievementService eduAchievementService;
}
```

- 方法定义的大括号也多为另起一行。
- 语句通常不省略花括号，尤其是在 `if`、`for`、`else` 结构中。
- import 顺序大致为：
  - JDK
  - `javax`
  - 第三方 / Spring
  - 项目内包
- 注释以中文为主，接口说明常用 Javadoc。

### 3.4 注释风格

- Controller、Service、Entity 基本都有类级 Javadoc。
- 方法前通常有简短中文注释说明用途。
- 业务判断内部常穿插步骤式注释，如“1. 先查旧数据”“2. 权限校验”。
- 新增业务代码中的注释明显比基础模板更详细，说明项目接受“解释业务规则”的注释，但不适合写成过度描述性的流水账。

### 3.5 Spring 使用习惯

- 依赖注入以字段注入 `@Autowired` 为主，当前项目中构造器注入不是主流。
- 校验能力在系统模块中有使用 `@Validated`，但业务模块使用并不完全统一。
- 全局异常处理集中在 `GlobalExceptionHandler`，业务接口多数直接返回 `AjaxResult`。
- 当前代码更偏向“显式判断 + 返回错误消息”，而不是大量抛出自定义异常。

### 3.6 实体类风格

- 不使用 Lombok，实体类手写字段、getter/setter、`toString()`。
- 常继承 `BaseEntity`，复用 `createBy`、`createTime`、`updateBy`、`updateTime` 等公共字段。
- 导出相关字段使用 `@Excel` 标注。
- 字段上常附带中文块注释，解释数据库含义和业务意义。

## 4. MyBatis XML 风格

- XML 中 SQL 关键字通常使用小写，如 `select`、`from`、`where`、`order by`。
- `resultMap`、公共 `sql` 片段、列表查询、详情查询、增删改语句的组织顺序较固定。
- 动态 SQL 常见写法：
  - `<where>`
  - `<if test="...">`
  - `<trim prefix="SET" suffixOverrides=",">`
  - `<foreach ...>`
- 字段映射显式书写，不依赖隐式驼峰映射。
- SQL 语义上偏直接、清晰，不追求复杂抽象。

## 5. 前端 Vue 风格

### 5.1 基础格式

- `edu-ui/.editorconfig` 明确约定：
  - UTF-8
  - 2 空格缩进
  - `LF` 换行
  - 文件结尾保留换行
  - 去除尾随空格
- Vue 文件通常使用单文件组件，结构为 `template`、`script`、`style`。
- 整体以 Options API 为主，不使用 Composition API。

### 5.2 命名与组织

- 组件名通常使用 PascalCase 或语义化英文，例如 `TeacherAchievement`、`PanelGroup`。
- API 文件采用“一个业务域一个文件”的方式，每个函数对应一个接口。
- 方法名偏动词导向，如 `getList`、`handleQuery`、`handleDelete`、`loadProgress`。
- 查询参数常统一命名为 `queryParams`，表单对象常命名为 `form`，弹窗开关常命名为 `xxxOpen`。

### 5.3 代码写法

- 业务页面大量采用 `data() { return { ... } }`、`created()`、`methods`、`computed` 的经典 Vue 2 写法。
- Promise 与 `async/await` 混用，但都属于项目可接受写法：
  - 常规增删改查页面偏 `then(...)`
  - dashboard 之类聚合页面更容易使用 `async/await`
- 页面方法名普遍沿用后台管理模板习惯：
  - `getList`
  - `handleAdd`
  - `handleUpdate`
  - `submitForm`
  - `resetQuery`
- 全局能力倾向通过 `Vue.prototype` 挂载，例如字典、下载、弹窗、工具方法。

### 5.4 模板与样式风格

- 模板层大量使用 Element UI 组件。
- 业务页面在模板层会保留较多中文注释，例如“统计卡片”“详情弹窗”“审核进度弹窗”。
- 样式上大量使用：
  - 圆角
  - 浅阴影
  - 渐变色
  - 卡片化布局
  - 轻量悬浮动画
- 样式块常直接写在页面底部，局部页面多使用 `scoped`。
- 对 Element 内部样式覆盖时使用 `>>>` 深度选择器。

## 6. 当前较稳定的风格约定

后续新增代码建议优先遵循以下约定：

- 后端继续沿用若依式分层和返回模型，不要在业务模块突然切成另一套架构。
- Java 代码保持 4 空格缩进，大括号另起一行。
- Service 接口继续保留 `I` 前缀。
- 实体类继续手写 getter/setter，不要在局部偷偷引入 Lombok。
- Controller 继续使用 `AjaxResult`、`TableDataInfo`、`BaseController` 这一套公共能力。
- MyBatis XML 保持小写 SQL 关键字与动态 SQL 写法。
- 前端继续使用 2 空格缩进、Options API、`queryParams/form/loading/open` 这套命名习惯。
- 前端 API 封装保持“一函数一接口”，不要把复杂业务逻辑堆进 `api` 文件。
- 页面视觉延续当前卡片化、浅色、微圆角的设计语言。

## 7. 当前存在的不统一点

这些现象在仓库中真实存在，后续开发最好择一收敛：

- 前端引号风格不完全统一，模板文件里单双引号混用。
- 前端分号使用不完全统一，有的文件偏模板原风格，有的业务页会大量使用分号。
- 后端注释密度不一致，基础模块较克制，新增业务模块注释明显更重。
- 个别业务接口命名不够统一，例如教师端接口既有 `teacherListAchievement`，也有 `teacherGetAchievement`、`teacherResubmit` 这类不同粒度命名。
- 个别注释与实现存在轻微偏差，例如注释提到逻辑删除，但 XML 实际执行的是物理删除，这类情况后续需要同步修正。

## 8. 推荐的后续执行原则

- 新代码优先模仿所在目录最近、最稳定的文件，而不是只参考 README。
- 修改现有模块时，尽量服从该模块原有风格，不做跨文件的大范围风格“纠正”。
- 如果后续要统一风格，建议先补充正式规范文件，再引入格式化或静态检查工具。
